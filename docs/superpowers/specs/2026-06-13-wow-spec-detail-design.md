# WoW 专精详情与 Portal 数据链路修复设计

## 背景

当前新增的 WoW 专精详情功能覆盖了人物榜、玩家详情、M+ 统计、团本统计，以及若干 Portal 首页榜单。实际复盘和本地验证后，发现这块能力同时存在数据正确性、运行可验证性和页面展示三类问题，其中最严重的是人物榜 Top20 与真实榜单不对应。

本设计的目标不是做抽象优化，而是围绕当前线上功能做一次可验证的修复：先把数据抓对，再把本地 Windows 验证链路跑通，最后重做天赋图与相关展示。

## 已确认事实

### 数据链路事实

1. `SeasonMeta` 当前活跃赛季为 `mn-s1`，`rio_season` 为 `season-mn-1`，与 Raider.IO 当前赛季一致，问题不在赛季 ID。
2. `SpecDetailPlayerMonitor._fetch_top_players()` 当前调用 `fetch_raiderio_top()` 时没有显式传 `page`，而 `fetch_raiderio_top()` 默认 `page=1`。
3. Raider.IO `mythic-plus/rankings/specs` 接口采用 0 基分页。`page=0` 才是前 20，`page=1` 实际是第 21-40 名。
4. 当前数据库中 `PlayerSpecTopPlayer` 的错误榜单，和 Raider.IO `EU/US page=1` 返回内容逐项一致，已确认“抓到的是各地区第 21-40 名，再合并收束”的根因。
5. `SpecStatsService.get_player_list()` 直接按 `rank` 排序输出，而当前 `rank` 是写入顺序，不是严格语义上的全球排名。

### 天赋展示事实

1. `PlayerSpecTopPlayer.talents_json` 存在两种格式：结构化节点数组，以及 `talentLoadoutText` 字符串包装成的数组。
2. `SpecDungeonRanking` / `SpecRaidRanking` 里的很多天赋节点只有 `spellID/talentID`，缺少 `name/icon`。
3. 当前页面无法构建类似 `murlok.io` 的树状天赋图，因为系统内没有稳定的“树节点元数据 + 节点坐标 + 节点归属（职业 / 专精 / 英雄）”。
4. 玩家详情页当前已退化成 `? spellID` 列表，说明这个问题不是纯 CSS，而是数据标准和组件形态都不成立。

### 运行与验证事实

1. 本机通过项目 `.venv` 可以启动 Django，并能直连开发数据库。
2. `settings_dev.py` 存在 MySQL 初始化 SQL 兼容问题，在 Windows 本机上会因为 `init_command` 多语句报错。
3. 现有 monitor 主要按 Linux 环境运行，但核心代码并不天然依赖 Linux；主要问题在配置、入口和验证手段不足，而不是算法无法在 Windows 运行。

## 问题定义

这次修复围绕三个问题展开：

1. 人物榜数据错误：Top20 抓取逻辑与展示语义不一致，且分页参数错误导致榜单完全偏移。
2. 本地无法稳定复现：开发配置和 monitor 本地执行入口不足，导致每次修改后很难做闭环验证。
3. 页面信息结构差：天赋展示没有稳定的数据标准支撑，UI 层只能输出退化信息。

## 目标

### 必须达成

1. 任一专精的人物榜 Top20 与 Raider.IO 当前世界榜逐项一致。
2. 本地 Windows 环境可以连开发库、运行 Django、执行指定 monitor，并验证改动结果。
3. 玩家详情页与统计页中的天赋展示不再出现 `? spellID`，而是可读、可视化、可聚合。
4. 每一轮代码改动都有明确验证，并按逻辑切分为独立 commit 后 push。

### 本轮不做

1. 不重构整个 LMonitor 调度系统。
2. 不把所有统计逻辑一次性改为离线预计算。
3. 不尝试在一轮内完全复刻 `murlok.io` 的所有视觉细节与全站功能。

## 方案概览

按四个阶段推进，每个阶段都要求本地验证、commit、push。

### 阶段一：修正人物榜抓取与展示语义

#### 修改点

1. 修正 `fetch_raiderio_top()` 的分页默认值，确保默认抓第一页而不是第二页。
2. `SpecDetailPlayerMonitor._fetch_top_players()` 改为优先使用 `region=world` 直接获取世界榜。
3. 多地区抓取保留为兜底策略，不再作为默认“全球榜”实现。
4. `PlayerSpecTopPlayer.rank` 的写入语义改为真实输出顺位，即最终榜单位置。
5. 页面和文案从“全球 Top 20”到榜单来源说明全部与真实数据语义保持一致。

#### 验证方式

1. 本地直接调用 monitor 的抓取函数，对比 Raider.IO `world page=0`。
2. 执行 monitor 写回开发库后，核对数据库前 20 条与实时接口逐项一致。
3. 打开本地页面，确认前 20 名与数据库一致。

### 阶段二：建立 Windows 本地验证链路

#### 修改点

1. 修正 `settings_dev.py` 中 MySQL 连接兼容问题。
2. 明确 Windows 本地启动与 Linux 线上执行的兼容边界，尽量收敛到配置层。
3. 增加便于本地验证的命令或脚本，支持：
   - Django 健康检查
   - 数据库连通检查
   - 指定 monitor 单次执行
   - 关键表结果校验

#### 验证方式

1. 本机使用 `.venv` 完成 `manage.py check`。
2. 本机完成开发库读写。
3. 本机手动执行相关 monitor 并看到结果更新。

### 阶段三：标准化天赋数据

#### 修改点

1. 统一 `talents_json` 的输出结构，不再允许混合字符串和结构化数组。
2. 对人物榜、WCL 明细、后续聚合统一使用同一种节点字段模型。
3. 为 UI 提供天赋聚合所需字段：
   - `tree_type`
   - `spell_id`
   - `talent_id`
   - `name`
   - `icon`
   - `points`
   - `row`
   - `column`
4. 增加“天赋节点元数据”来源，补齐至少一套可渲染的静态树定义能力。
5. 服务层增加节点使用率聚合输出，而不是只返回原始点法列表。

#### 验证方式

1. 任一专精都可得到统一格式的天赋节点集合。
2. 玩家详情页天赋数据不再出现无名节点。
3. 统计页可以输出节点使用率。

### 阶段四：重做 Portal 专精详情 UI

#### 页面结构

1. 顶部概览：专精图标、赛季、样本量、更新时间。
2. 榜单卡片：真实 Top20，突出排名、地区、分数。
3. 天赋热力图区：
   - 职业树
   - 专精树
   - 英雄天赋树
   - 节点图标 + 使用率
4. 常见构筑摘要：按样本聚合主流 build。
5. 装备统计卡：饰品、套装、主手、戒指等高频项。
6. M+ / 团本统计卡：保留既有统计结果，但提升版式和信息优先级。

#### 视觉原则

1. 信息卡片化，统一间距和色板。
2. 天赋图区优先展示图标和占比，不再以纯文字为锚点。
3. 视觉目标参考 `https://murlok.io/monk/windwalker/m+#top-ratings`，但先保证信息结构正确，再逐步贴近观感。

#### 验证方式

1. 本地页面可访问。
2. 页面内容与数据库/服务层结果一致。
3. 天赋区域可以一眼看出“点了什么、占比多少、属于哪棵树”。

## 关键设计决策

### 为什么优先修数据而不是直接改 UI

当前最显著的问题虽然体现在页面上，但核心故障出在数据层。若不先修正榜单抓取和天赋结构，任何 UI 重做都只是把错误数据包装得更漂亮。先保证数据正确，才能让后续样式和组件有稳定输入。

### 为什么人物榜改为优先读 `world`

当前“多地区抓取再合并”的近似方案理论上可用，但需要自己承担区域拼接、去重、排名语义等额外复杂性。既然 Raider.IO 已直接提供 `world` 榜单，默认应直接使用其世界榜，只有在接口异常时才考虑多地区兜底。

### 为什么需要天赋树元数据

仅靠当前抓到的 `talentID/spellID` 和部分 `name/icon`，不足以还原出树状布局。若要实现近似 `murlok` 的体验，必须让服务层或前端知道每个节点位于哪一棵树、哪一行、哪一列，否则最多只能做图标列表，无法做真正的天赋图。

## 风险与控制

### 风险

1. 外部接口字段可能继续变化，尤其是 Raider.IO 天赋与 gear 结构。
2. 英雄天赋和部分专精的树定义可能无法从现有数据源稳定获得。
3. 页面重做后需要兼顾现有模板和全站 Portal 样式，不宜一次性改太大。

### 控制方式

1. 先针对一个专精完成全链路验证，再推广到全职业。
2. 所有关键外部接口响应结构，都通过本地采样验证后再落库。
3. 对天赋图先落地“可读且正确”的一版，再做视觉增强。

## 验收标准

### 数据验收

1. 抓取函数输出与 Raider.IO 世界榜一致。
2. 数据库中人物榜前 20 与实时榜逐项一致。
3. 不再出现 `page=1` 偏移导致的榜单错位。

### 功能验收

1. Windows 本地可执行相关 monitor。
2. 专精详情页可正常访问、列表可跳转、详情可展示。
3. 天赋、装备、统计模块都能稳定显示。

### 展示验收

1. 页面结构清晰，不再是简单列表堆叠。
2. 天赋图以图标和使用率为核心表达。
3. 用户无需查看原始 JSON 即可理解当前主流构筑。

## 实施与提交流程

后续执行遵循固定流程：

1. 每次只处理一个明确子问题。
2. 本地完成数据与页面验证。
3. 形成独立 commit。
4. push 到远程。

建议的提交分组：

1. 修正榜单抓取与验证脚本
2. 修正本地开发配置与 monitor 执行入口
3. 统一天赋数据结构与服务层聚合
4. 重做 Portal 页面与天赋图展示
