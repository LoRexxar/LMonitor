# WoW 通用天赋树渲染引擎设计

## 背景

当前玩家详情页里的天赋展示已经从纯 ID 列表提升到了“按树分组 + 节点坐标”的层级，但离真实游戏天赋图仍有明显差距。主要问题不在单一 CSS 细节，而在底层数据与渲染结构仍然偏页面拼装：

1. 页面直接消费 `talent_groups` 和 `group.nodes` 这类贴近模板的数据结构，缺少一个通用的树模型。
2. 当前节点虽然已有 `tree_type / row / column`，但没有稳定的“边关系 / 前置关系 / 选择节点 / 节点形态”表达。
3. 页面布局仍以 `CSS Grid + 卡片` 为主，能摆出节点，但还原不了原版天赋图的主干连线、分支走向和视觉层次。
4. 副本/团本页与玩家页分别消费不同粒度的数据，未来如果要支持热门天赋树、双 build 对比、导入码映射，会继续在服务层和模板层重复造结构。

这轮设计不再只修“当前这一个图”，而是要建立一个可以扩展到各种天赋树图场景的通用底座，并以玩家详情页为第一落地点。

## 目标

### 必须达成

1. 建立一套独立于页面模板的通用天赋树模型，能同时表达职业树、专精树、英雄树。
2. 将“树定义”和“build 状态”分离，让玩家 build、热门天赋、差异高亮都能复用同一棵树定义。
3. 采用适合树状结构的渲染方式，让页面视觉更接近原版天赋图，而不是继续堆卡片。
4. 第一阶段完成玩家详情页接入；第二阶段复用到底层，支持副本/团本热门天赋展示；第三阶段预留对比与差异视图能力。
5. 保持与当前数据库、monitor、`build_talent_view_model(...)` 兼容，先通过 adapter 过渡，不强行重写全链路。

### 本轮不做

1. 不一次性复刻游戏客户端的所有视觉细节、动画和拖拽体验。
2. 不在本轮引入完整的客户端级交互状态机，例如点天赋、撤销、校验点数合法性。
3. 不在 monitor 落库阶段立刻改造为完整树定义存储；优先完成展示引擎与服务层重构。

## 已确认事实

1. `PlayerSpecTopPlayer.talents_json` 已能稳定保存结构化节点，`build_talent_view_model(...)` 可输出 `trees`、`nodes`、`build_code`。
2. `WowTalentNodeMetadata` 及相关回填命令已经能提供 `tree_type / row / column / icon / spell_id` 等关键元数据。
3. 当前玩家详情页模板 `player_detail.html` 已经按 `group.grid_columns / group.grid_rows / node.layout_row / node.layout_column` 渲染。
4. 当前方案的问题主要集中在：
   - 节点模型仍偏展示结果；
   - 缺少边与节点形态定义；
   - 渲染方式上限较低；
   - 不利于多场景复用。

## 设计原则

### 定义与状态分离

树本身的结构定义必须和某次展示的点法状态解耦：

1. 树定义回答“这棵树本来长什么样”。
2. build 状态回答“这次展示里哪些节点被点了、点了几层、如何高亮”。

只有这样，玩家详情、热门统计、导入码对比才不会各自产一套树。

### 布局与渲染分离

服务层不直接返回“模板可用的最终 HTML 形状”，而是返回：

1. 结构化树定义
2. 状态
3. 布局结果
4. 渲染模型

模板只负责消费渲染模型，不再在模板里推导坐标、节点类型和树分组。

### 先兼容，再替换

优先通过 adapter 接入现有 `talents_json` 和元数据表，不强求 monitor 先改输出格式。底层稳定后，再逐步把 monitor/聚合层也收敛到新模型。

## 架构分层

建议把天赋图相关能力拆成四层。

### 原始层

职责：

1. 读取 `talents_json`
2. 读取元数据表
3. 归一化图标、`spell_id`、`talent_id`
4. 兼容历史落库格式

代表输入：

1. `PlayerSpecTopPlayer.talents_json`
2. `SpecDungeonRanking.talents_json`
3. `SpecRaidRanking.talents_json`
4. `WowTalentNodeMetadata`

这一层不关心页面结构。

### 语义层

职责：

1. 将原始节点转换成通用树模型
2. 把职业/专精/英雄树组织成统一的 `TalentTreeSet`
3. 表达节点定义、边定义、节点状态

这是整个方案的核心层，也是后续所有页面复用的基础。

### 布局层

职责：

1. 根据树定义和布局策略计算节点在画布中的位置
2. 生成连线路径
3. 处理三棵树并排或堆叠的容器布局
4. 处理移动端降级布局

布局层输出的是坐标系结果，不直接拼 HTML。

### 渲染层

职责：

1. 生成模板所需的统一 `render_model`
2. 节点用 HTML 渲染
3. 连线和高亮层用 SVG 渲染

渲染层对上层页面保持同一接口，避免玩家页和统计页各写一套树模板逻辑。

## 通用数据模型

### TalentTreeSet

表示一次完整展示的树集合，例如某专精的职业树、专精树、英雄树。

建议字段：

1. `set_key`
2. `class_name`
3. `spec_name`
4. `trees`
5. `layout_mode`
6. `meta`

作用：

1. 描述当前这一屏要展示几棵树
2. 控制树顺序和分组方式
3. 作为玩家页、统计页、对比页的统一入口对象

### TalentTreeDefinition

表示单棵树的静态定义。

建议字段：

1. `tree_id`
2. `tree_type`：`class / spec / hero / other`
3. `title`
4. `subtitle`
5. `theme`
6. `grid.columns`
7. `grid.rows`
8. `nodes`
9. `edges`

说明：

1. 这里不放“这个角色点了几层”的动态信息。
2. 这一层尽量贴近游戏树结构，便于后续高保真渲染。

### TalentNodeDefinition

表示节点本体的静态信息。

建议字段：

1. `node_key`
2. `spell_id`
3. `talent_id`
4. `name`
5. `icon`
6. `row`
7. `column`
8. `max_ranks`
9. `node_kind`
10. `shape_hint`
11. `choice_group`
12. `tree_type`

说明：

1. `node_kind` 至少支持普通节点、选择节点、关键节点、大招节点。
2. `shape_hint` 用于后续渲染层决定圆形、八边形、强调边框等视觉样式。
3. `choice_group` 用于表达互斥选择节点。

### TalentEdgeDefinition

表示节点之间的结构关系。

建议字段：

1. `edge_key`
2. `from_node`
3. `to_node`
4. `edge_kind`
5. `path_hint`
6. `is_required`

说明：

1. `edge_kind` 至少支持主干、分支、互斥选择关联。
2. `path_hint` 用于布局层生成直线、折线或曲线路径。
3. 没有这层结构，就无法真正接近原版天赋图。

### TalentBuildState

表示某个角色或某组统计在树上的动态状态。

建议字段：

1. `source_type`：`player / dungeon_usage / raid_usage / comparison`
2. `source_id`
3. `selected_nodes`
4. `node_ranks`
5. `choice_selection`
6. `heat_values`
7. `highlight_groups`
8. `build_code`
9. `confidence`

说明：

1. 玩家详情页主要关注 `selected_nodes / node_ranks / build_code`。
2. 副本/团本热门天赋页主要关注 `heat_values`。
3. 对比页未来主要关注 `highlight_groups` 和多状态并置。

### TalentRenderModel

这是模板直接消费的对象，来自“树定义 + 状态 + 布局”的组合结果。

建议字段：

1. `canvas`
2. `tree_panels`
3. `rendered_nodes`
4. `rendered_edges`
5. `legend`
6. `build_summary`

说明：

1. 页面模板不再直接依赖原始 `talents_json`。
2. 页面模板只渲染 `rendered_nodes` 和 `rendered_edges`。

## 渲染方案选择

比较三种候选方案：

### 方案一：纯 DOM / CSS

优点：

1. 与当前模板体系最兼容
2. 节点 hover、跳转、tooltip 简单

缺点：

1. 连线、分支、前置关系表达困难
2. 上限偏低，长期仍会变成模板补丁

### 方案二：DOM + SVG 混合

优点：

1. 节点仍是 HTML，交互和可维护性最好
2. 连线、路径、高亮层交给 SVG，最适合树结构
3. 与 Django 模板和现有页面体系兼容度高
4. 便于后续支持统计热度覆盖、差异高亮、双 build 叠加

缺点：

1. 需要先把节点坐标与边关系抽象清楚
2. 初期实现比纯 DOM 稍复杂

### 方案三：纯 SVG / Canvas

优点：

1. 自由度最高
2. 理论上最适合高度图形化场景

缺点：

1. 文本、交互、模板接入复杂度更高
2. 第一阶段过重，不适合作为本轮主方案

### 结论

采用 `DOM + SVG 混合` 作为正式方案：

1. 节点层：HTML
2. 连线层：SVG
3. 高亮 / 热度 / 差异辅助层：SVG + CSS

这是当前最适合“接近原版天赋图，同时保持可扩展性”的方案。

## 模块拆分

建议新增或重构以下模块。

### `botend/wow/talents/models.py`

职责：

1. 定义通用树对象的数据结构
2. 统一字段命名和类型

建议包含：

1. `TalentTreeSet`
2. `TalentTreeDefinition`
3. `TalentNodeDefinition`
4. `TalentEdgeDefinition`
5. `TalentBuildState`
6. `TalentRenderModel`

### `botend/wow/talents/adapters.py`

职责：

1. 从当前 `talents_json` 和元数据表构造新模型
2. 向后兼容历史落库格式

### `botend/wow/talents/layout.py`

职责：

1. 布局策略
2. 节点坐标与树面板尺寸计算
3. 连线路径生成
4. 移动端降级布局

### `botend/wow/talents/render.py`

职责：

1. 将定义、状态、布局结果组合成模板可直接消费的 `render_model`
2. 输出节点外观字段、边路径字段、图例字段

### `botend/wow/talents/view_model.py`

职责调整：

1. 保留旧入口兼容
2. 内部逐步改为调用 adapter + layout + render
3. 让现有服务层不必一次性重写所有调用点

## 页面接入策略

### 第一阶段：玩家详情页

接入目标：

1. 将 `player_detail.html` 的天赋区切换为统一树渲染模型
2. 三棵树并排展示
3. 展示节点形态、点数、前置连线
4. 保留 hover 浮层和 Wowhead 跳转

验收标准：

1. 明显比当前卡片网格更接近原版结构
2. 不再依赖模板内自行拼装行列信息

### 第二阶段：副本 / 团本热门天赋

接入目标：

1. 统计页复用同一棵树定义
2. 用热度覆盖替代单纯 Top N 节点列表
3. 能看出高频点法集中在哪些区域

验收标准：

1. 统计页和人物页共用同一底层模型
2. 不再单独维护两套天赋展示逻辑

### 第三阶段：差异与对比

接入目标：

1. 支持两个 build 的差异高亮
2. 支持导入码映射后的并置展示

这一阶段先预留结构，不作为本轮必须完成内容。

## 数据兼容与过渡

### 过渡策略

本轮不直接修改 monitor 落库结构，而是增加 adapter。

输入：

1. `talents_json`
2. `WowTalentNodeMetadata`
3. 当前 `build_talent_view_model(...)` 已能拿到的字段

输出：

1. 通用树定义
2. build 状态
3. 渲染模型

### 后续收敛方向

等渲染引擎稳定后，再考虑：

1. monitor 直接落更完整的树状态
2. 统计聚合直接基于新模型生成热门树状态

这样可以降低本轮改造风险。

## 测试策略

### 单元验证

1. adapter 是否能把现有 `talents_json` 转成树定义与状态
2. 布局层是否能稳定生成节点坐标与边路径
3. 渲染层是否能在同样输入下生成稳定 `render_model`

### 页面验证

1. 玩家详情页：验证三棵树的顺序、坐标、节点激活状态、连线
2. 副本 / 团本页：验证热门节点热度映射与树结构复用
3. 移动端：验证树在窄屏下是否降级为可读布局

### 回归验证

1. 不破坏现有 `build_talent_view_model(...)` 的其他消费者
2. 不影响当前 monitor、数据落库与统计接口
3. 本地页面不出现新的外链图标、静态资源错误或布局阻塞问题

## 风险与控制

### 风险

1. 当前元数据里边关系可能不完整，导致连线恢复度不够。
2. 某些专精或英雄树的节点形态信息不足，初期可能仍需降级样式。
3. 如果一次性替换所有页面，回归范围会过大。

### 控制

1. 先用玩家详情页验证新引擎，再扩到统计页。
2. 节点形态和边关系缺失时，允许优雅降级为“无连线 / 默认节点形态”，但模型字段必须预留。
3. 保留旧 `view_model` 兼容层，分阶段切换页面。

## 分阶段实施

### 阶段一：建立通用模型与 adapter

1. 新增通用对象定义
2. 新增 adapter
3. 保持旧接口可用

### 阶段二：建立布局与 SVG 渲染层

1. 实现 `DOM + SVG` 混合树渲染
2. 输出统一 `render_model`
3. 处理树面板、节点层、连线层

### 阶段三：接入玩家详情页

1. 替换当前卡片网格布局
2. 复用已有 hover、跳转、图标能力
3. 完成页面验证

### 阶段四：接入统计页

1. 用同一引擎展示热门天赋树
2. 将热度映射到节点状态
3. 清理旧 Top N 节点列表式展示逻辑

### 阶段五：预留对比能力

1. 在模型里固化差异高亮字段
2. 不一定在本轮完成 UI，但要保证底层结构已支持

## 验收标准

### 架构验收

1. 页面不再直接依赖 `talent_groups` 这种页面专用结构作为最终形态。
2. 玩家页和统计页共用一套树定义与渲染底座。
3. 定义、状态、布局、渲染四层边界清晰。

### 展示验收

1. 玩家详情页天赋图明显接近原版三树布局。
2. 节点不再只是卡片网格，而是具备树状关系表达。
3. 节点 hover、点数显示、链接跳转保持可用。

### 扩展验收

1. 新增一个树图消费者时，不需要再复制一套模板和服务层逻辑。
2. 热门天赋、单角色 build、未来差异视图都可以复用同一底层模型。
