# WoW 天赋字符串统一存储与解释链路重构设计

## 背景

当前 WoW 天赋树模块已经经历过一次拆分与重做，目标原本很清晰：

1. 先保存和处理原始天赋字符串。
2. 再把字符串解释为天赋节点树。
3. 最后由前端模块化展示。

但实际落地后，链路重新发生了耦合：

1. `PlayerSpecTopPlayer`、`SpecDungeonRanking`、`SpecRaidRanking` 等表里，主存储并没有稳定保存原始天赋字符串，而是保存了节点列表。
2. 采集阶段拿到的 Raider.IO `talentLoadoutText`，在标准化落库过程中被丢失。
3. `view_model/render_model` 这类展示层结构，被反向用于采集和落库中间层，导致“解释层”和“展示层”边界被打穿。
4. 详情页能展示天赋树，并不是因为数据库里保存了原始字符串，而是因为后端直接返回了结构化节点和渲染模型。

这一轮不再做局部修补，而是把天赋树系统拉回你要求的三段式结构：字符串是唯一真相源，解释层统一复用，API 聚合前端数据，前端只做展示。

## 已确认事实

### 字符串来源事实

1. `SpecDetailPlayerMonitor._parse_rio_talents()` 会从 Raider.IO 的 `talentLoadoutText` 读取原始字符串。
2. 该函数会先构造一条 `build_code` 节点，包含 `talent_code`。
3. 当前并不是“源头没有字符串”，而是在后续标准化过程中丢失了字符串。

### 丢失点事实

1. `SpecDetailPlayerMonitor._normalize_talent_nodes()` 会调用 `build_talent_view_model(...)`。
2. 这个函数随后只把 `view_model['nodes']` 重新写入数据库。
3. `build_code` 并不在 `nodes` 里，而在 `view_model['build_code'] / render_model['build_code']`。
4. 因此 `talentLoadoutText` 在落库时被静默丢弃，最终数据库只剩节点列表。

### 存储现状事实

1. `PlayerSpecTopPlayer.talents_json` 当前保存的是节点列表，而不是原始天赋字符串。
2. `SpecDungeonRanking.talents_json` 与 `SpecRaidRanking.talents_json` 也以节点列表为主。
3. 这些节点列表足以支撑当前的树渲染，但不符合“字符串单一真相源”的架构约束。

### 业务约束

这轮设计需要严格满足以下约束：

1. 所有相关存储位置都必须以天赋字符串作为主存储。
2. 允许没有天赋树字符串，空值是合法状态。
3. 必须通过脚本直接完成服务器脏数据清理和替换。
4. 所有涉及天赋树的模块都必须复用同一个解释类。
5. API 只负责调用解释类并聚合前端数据。
6. 前端使用专门的 JS 模块和 API 配置完成展示。
7. 可以加入缓存，但缓存不能违背“字符串主存储、运行时派生”的基础逻辑。
8. `monitor` 及相关刷新命令必须同步更新，不能继续按旧节点链路写脏数据。

## 问题定义

当前问题不是单一字段缺失，而是底层契约错位：

1. 存储层存错了东西。数据库主存储保存的是“解释结果”，而不是“原始输入”。
2. 解释层和展示层发生反向依赖。展示模型被拿去参与落库标准化。
3. monitor 链路继续写入节点型主数据，会不断污染新结构。
4. 历史脏数据已经存在于生产数据中，必须通过脚本清理和迁移，而不是等待新采集慢慢覆盖。

## 目标

### 必须达成

1. 所有相关业务表的天赋主存储统一为 `talent_build_code` 字符串字段。
2. 缺失字符串时允许为空，不伪造、不反推。
3. 建立统一解释类，所有天赋树相关模块只能通过它解析字符串、构建树和生成 render model。
4. API 层统一从字符串生成前端数据，不再直接读取节点列表作为真相源。
5. `monitor`、刷新命令、服务层、详情页、统计页同步改造到新链路。
6. 提供可直接执行的脏数据清理脚本，完成旧数据迁移与净化。
7. 详情页显示导入代码字符串，并提供复制图标按钮，点击后切换为 `已复制`。

### 本轮允许

1. 某些记录没有天赋字符串。
2. 解析结果可以缓存。
3. 历史遗留的 `talents_json` 字段可以短期保留作为过渡字段。

### 本轮不做

1. 不从节点列表反推或伪造导入字符串。
2. 不让前端负责解释天赋字符串。
3. 不把缓存当成主数据来源。
4. 不继续扩散“节点列表主存储”的旧模式。

## 核心设计原则

### 字符串单一真相源

数据库里关于天赋的主数据，唯一可信输入是 `talent_build_code`。

这意味着：

1. 任何树结构、节点关系、render model 都是派生结果。
2. 业务代码不得再把节点列表当成事实源。
3. API 返回的树与前端看到的树，必须能追溯到同一条字符串。

### 解释链路单入口

所有与天赋树相关的逻辑，统一经由一个解释服务完成，而不是：

1. 有的页面调 `parser`
2. 有的服务调 `view_model`
3. 有的 monitor 直接拼节点
4. 有的统计直接读 `talents_json`

只有统一入口，才能避免未来再次发生结构漂移。

### 存储层与展示层彻底解耦

数据库只保存原始字符串，不保存前端展示形状。前端只消费 API 返回的展示结果，不再知道数据库内部怎么存。

### 缓存只是派生加速器

缓存的存在不能改变真相源，不允许出现“数据库没字符串，但缓存还能凑出一棵树”的逆向依赖。

## 目标架构

重构后固定为四层。

### 第一层：字符串存储层

职责：

1. 保存 `talent_build_code`
2. 表达“有字符串 / 无字符串”的基础状态
3. 不承担解释和展示职责

适用表：

1. `PlayerSpecTopPlayer`
2. `SpecDungeonRanking`
3. `SpecRaidRanking`

建议契约：

1. `talent_build_code` 为主字段，允许空字符串或空值。
2. `talents_json` 作为历史过渡字段保留，但不再作为主读取来源。

### 第二层：统一解释层

职责：

1. 接收字符串、职业、专精
2. 校验字符串是否存在
3. 解析字符串并转换为标准 payload
4. 构建树结构
5. 生成渲染模型

该层输出的是内存对象，而不是数据库字段。

### 第三层：API 组装层

职责：

1. 从数据库读取 `talent_build_code`
2. 调用解释层生成结构
3. 聚合为页面需要的接口数据
4. 明确区分“无字符串”“解析失败”“解析成功”

### 第四层：前端展示层

职责：

1. 通过专门的 JS 模块消费 `talent_render_model`
2. 渲染天赋树
3. 渲染导入代码和复制交互
4. 不解释字符串，不推导节点，不参与业务数据纠正

## 数据契约

### 主字段

统一新增或使用：

1. `talent_build_code`

建议语义：

1. 类型：字符串
2. 可为空
3. 含义：原始天赋导入字符串
4. 真相源级别：唯一主输入

### 旧字段

`talents_json` 进入过渡期：

1. 允许暂时保留
2. 不再作为新逻辑的主依赖
3. 新 monitor 不再把它当主写入目标
4. 后续稳定后可逐步降级为辅助调试字段，甚至移除

### 空值规则

1. 没字符串是合法状态
2. 解释层必须能处理空字符串
3. 页面必须能优雅降级
4. 不允许为了“页面能显示”而伪造字符串

## 统一解释类设计

建议新增统一入口类：

`TalentBuildCodeService`

### 对外接口

建议提供四类方法：

1. `parse_build_code(build_code, class_name, spec_name)`
2. `build_tree(build_code, class_name, spec_name)`
3. `build_render_model(build_code, class_name, spec_name)`
4. `build_api_view(build_code, class_name, spec_name)`

### 责任边界

这个类内部可以复用现有：

1. `parser`
2. `adapter`
3. `render`
4. `view_model`

但这些模块不再被业务层随意直接调用。

### 返回约定

解释类必须统一处理三种状态：

1. 有字符串且解析成功  
   返回完整 payload、tree、render model

2. 有字符串但解析失败  
   返回错误状态和空展示结果

3. 没有字符串  
   返回空状态，不再偷偷回退到旧节点存储

## API 设计

### 详情页接口

详情页建议固定返回：

1. `talent_build_code`
2. `has_talent_build_code`
3. `talent_parse_status`
4. `talent_render_model`

### 返回规则

1. 有字符串且解析成功：
   - 返回字符串
   - 返回 render model
   - `talent_parse_status = success`

2. 有字符串但解析失败：
   - 返回字符串
   - render model 为空
   - `talent_parse_status = failed`

3. 没有字符串：
   - `talent_build_code = ''`
   - render model 为空或降级
   - `talent_parse_status = missing`

### 统计页与热门天赋页

副本/团本/热门天赋页也统一走解释类，不再直接把数据库里的 `talents_json` 当主输入。

## 前端设计

### 详情页天赋代码展示

位置：

1. 详情页“天赋配置”卡片的代码区域

行为：

1. 有字符串时：
   - 展示 `talent_build_code`
   - 展示复制图标按钮
   - 点击后按钮状态切换为 `已复制`

2. 无字符串时：
   - 展示 `暂无导入代码`
   - 不显示复制按钮

### 前端模块边界

前端拆成两个专用模块：

1. 天赋树渲染 JS 模块  
   只消费 `talent_render_model`

2. 天赋代码交互模块  
   负责复制按钮状态切换

前端不负责：

1. 解释字符串
2. 读取旧节点格式
3. 拼接派生结构

## Monitor 与刷新命令改造

这是本次设计的硬要求，不能遗漏。

### monitor 新规则

`SpecDetailPlayerMonitor` 及相关命令必须改为：

1. 采集时优先读取 `talentLoadoutText`
2. 直接写入 `talent_build_code`
3. 不再通过 `_normalize_talent_nodes()` 将字符串转换为节点主存储
4. 不再让 `build_talent_view_model()` 参与落库标准化

### 适用范围

必须同步改造：

1. 人物榜采集 monitor
2. `refresh_spec_detail_players` 命令
3. 任何刷新玩家详情的命令
4. 副本 ranking 采集
5. 团本 ranking 采集
6. 任何写入天赋数据的脚本

### 结果要求

monitor 以后只负责：

1. 获取字符串
2. 落库存字符串

monitor 不再负责：

1. 固化节点树
2. 把展示层输出回写数据库

## 脏数据清理与服务器脚本

这部分必须直接执行，不是建议项。

### 目标

1. 识别当前所有旧节点型主存储记录
2. 迁移能保留下来的真实字符串
3. 将主读取链路切换到字符串字段
4. 清理或降级旧节点数据的主存储地位

### 脚本分两步

#### 第一步：迁移脚本

职责：

1. 在三张主要业务表中扫描历史记录
2. 尝试从旧字段中提取真实 `build_code / talent_code`
3. 写入 `talent_build_code`
4. 记录迁移统计

规则：

1. 能提取就迁移
2. 提取不到就留空
3. 不允许从节点列表反推伪造字符串

#### 第二步：净化脚本

职责：

1. 把旧节点型主存储从主链路中移除
2. 让新代码彻底不依赖旧节点字段
3. 输出剩余脏数据规模

### 执行要求

1. 脚本在服务器上直接执行
2. 执行前先统计
3. 执行后再统计
4. 明确输出迁移成功、留空、无法恢复的记录数

## 缓存设计

缓存允许加入，但只能做派生层缓存。

### 可缓存对象

1. 标准 payload
2. tree_set
3. render_model
4. API 聚合结果

### 缓存 key

建议由以下维度组成：

1. `class_name`
2. `spec_name`
3. `talent_build_code`
4. `engine_version`

### 缓存实现顺序

第一阶段：

1. 进程内 LRU 缓存

第二阶段：

1. 如有需要再扩展到 Redis

### 缓存硬边界

1. 缓存 miss 时必须回到字符串重新解释
2. 缓存不能替代数据库里的 `talent_build_code`
3. 缓存不能反向写数据库
4. 没有字符串时不能靠缓存伪造解析结果

## 迁移顺序

建议执行顺序如下：

1. 新增 `talent_build_code` 字段并补 migration
2. 落地 `TalentBuildCodeService`
3. 改详情页 API 和页面展示
4. 改所有 monitor / refresh 命令写入逻辑
5. 编写并执行服务器脏数据迁移脚本
6. 编写并执行净化脚本
7. 改统计页与热门天赋页到统一解释链
8. 观察稳定后，逐步切断旧 `talents_json` 的主依赖

## 风险与控制

### 风险

1. 部分历史数据无法恢复原始字符串。
2. 统计页目前可能存在对旧 `talents_json` 的隐式依赖。
3. monitor 链路改造不完整会持续生成新脏数据。
4. 运行时解释字符串可能带来额外开销。

### 控制方式

1. 明确接受“允许没有字符串”。
2. 所有读取统一走解释类，避免局部漏改。
3. monitor 改造与脏数据清理同步推进。
4. 用派生缓存抵消运行时解释成本。

## 验收标准

### 存储验收

1. 三张核心表的主存储都以 `talent_build_code` 为真相源。
2. monitor 新写入记录时，字符串不再丢失。
3. 不再出现“数据库只有节点，没有字符串，但逻辑仍当它是主数据”的情况。

### 详情页验收

1. 有字符串时显示导入代码。
2. 复制图标可用，点击后变为 `已复制`。
3. 无字符串时显示 `暂无导入代码`。

### 架构验收

1. API 不再直接从旧节点存储生成主展示结果。
2. 所有天赋树相关模块统一复用解释类。
3. 前端只消费 API 与 JS 模块，不承担解释职责。

### 清理验收

1. 服务器清理脚本执行成功。
2. 能迁移的字符串已迁移。
3. 不能恢复的记录已明确置空。
4. 清理后不再有新 monitor 持续写入旧脏格式。

## 实施结果预期

重构完成后，整个系统回到稳定边界：

1. 数据库存字符串
2. 解释层从字符串构建树
3. API 聚合前端模型
4. 前端只展示
5. monitor 只负责采集和落库存字符串
6. 缓存只负责加速派生结果

这会让详情页复制代码、玩家树渲染、热门天赋统计、后续构筑分析都回到同一条可维护链路上，不再靠历史节点存储勉强拼装。
